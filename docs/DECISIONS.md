# Architecture Decision Records

## 2026-07-09 — Gate Spans Can Mask Root-Cause Detection

**Bug:** `find_root_cause_span` (`src/analysis/root_cause.py`) stops the backward walk at the first (walking backward) span the step-quality judge scores healthy. `score_confidence` and `build_fallback_response` (`src/generation/confidence_scorer.py`, `src/generation/fallback_response.py`) are both `@traced("generation", ...)` deterministic, judge-free functions that mechanically combine already-computed upstream signals (`hits`, `citation_results`) — internally self-consistent by construction, so a step-quality judge scores them "reasonable" regardless of whether the upstream retrieval/ranking data they received was already corrupted. `traced()` auto-serializes a wrapped function's full bound arguments as `Span.input`, so the corrupted `hits` are literally present in these spans' input — but both use `step="generation"`, the same step a real answer-generation span would use, so they're judged against `STEP_QUALITY_CRITERIA["generation"]`, written for auditing an *answer* ("fully supported by context"), not for sanity-checking raw upstream data embedded in a serialized argument blob. Since these two spans sit near the end of most traces (`score_confidence` always runs; `build_fallback_response` runs on the low-confidence path), they could prematurely terminate the backward walk before it ever reached a genuinely corrupted upstream span — misattributing root cause to the gate span itself, or (if it's the trace's last span) returning `None`, "nothing wrong," on a visibly broken trace. Reproduced by synthetically corrupting a retrieval span (swapping in irrelevant chunks) and observing the walker fail to find it, landing instead on the fallback-decision span or `None`.

**Fix: a structural `Span.is_gate` flag, not prompt engineering** — Adding instructions to `STEP_QUALITY_CRITERIA["generation"]` telling the judge to sanity-check the `hits`/`citation_results` embedded in `input` was rejected: unreliable (depends on the judge noticing corrupted data buried in a blob it wasn't specifically asked to inspect for that purpose), and it would blur `"generation"`'s criteria — written for judging a real answer-generation span — with a second, unrelated auditing responsibility. It also contradicts the deliberate, documented "judge only this step's own transformation quality, not whether an earlier or later step in the pipeline was at fault" instruction in `ROOT_CAUSE_JUDGE_SYSTEM_PROMPT_TEMPLATE`, which is otherwise sound — for genuine pipeline-transformation spans (retrieval, ranking, real generation, verification), local-only judgment is correct, and each step's own `STEP_QUALITY_CRITERIA` already asks about query/evidence relevance where appropriate (retrieval's criteria explicitly requires "topically relevant to the query," ranking's requires "keeps... the chunks most relevant to the query"), so those steps self-report corruption correctly if the walk actually reaches them. Instead, `Span` gained `is_gate: bool = False`, threaded through a new `traced(..., is_gate=False)` parameter (default `False`, fully backward compatible — no existing `@traced` call site or `Span(...)` construction needed to change). `score_confidence`/`build_fallback_response` are the only two call sites that set `is_gate=True`. `find_root_cause_span` skips `is_gate=True` spans entirely: no judge call, no `evaluated_spans` entry, no candidate update, walk continues to the previous span — gate spans cost zero judge calls and can never terminate the walk or be misattributed as root cause.

**A new `PipelineStep` value was considered and rejected** — Giving gate spans their own step (e.g. `"gate"`) would let `find_root_cause_span` filter on `step` instead of a new field, but it unnecessarily touches `STEP_TO_PLAUSIBLE_CATEGORIES` (`failure_categorizer.py`) and every step-exhaustiveness test (`test_all_pipeline_steps_accepted`, etc.) for no benefit — gate spans should never become `root_cause_span` in the first place, so they don't need a place in the failure taxonomy at all. A boolean flag orthogonal to `step` says exactly what's needed without touching that surface.

**The judge's own inner span is not marked `is_gate`** — A real `CompletenessJudgeProtocol` provider opens its own `span("generation", ...)` inside `judge()` when `score_confidence` calls it; that inner span is a separate `Span` object reflecting a genuine LLM judgment call (is the answer complete?), not a deterministic pass-through of upstream data. Only the outer wrapper span `@traced` builds around `score_confidence`/`build_fallback_response` themselves is `is_gate=True` — if the walk reaches the judge's inner span, judging it against `"generation"` criteria is legitimate.

**`test_only_contiguous_unhealthy_tail_is_judged` is unchanged** — That test's "an earlier bad span behind an intervening healthy span is invisible" behavior is legitimate and intended for real pipeline-transformation spans: a genuinely-fixed intervening step really does mark a healthy boundary. This fix only changes behavior for spans that are structurally incapable of ever legitimately signaling a healthy boundary.

## 2026-07-09 — Leading-Marker Handling in `parse_citations`

**Bug:** `parse_citations` (`src/generation/citation_parser.py`) unconditionally took the text *preceding* a `[N]` run as the claim it supports. `GROUNDED_SYSTEM_PROMPT` never said which side of the claim the marker belonged on, so nothing stopped a model from opening a sentence with the marker instead — e.g. "According to the context provided, [1] the on-call rotation is weekly…". The parser then handed the judge the fragment "According to the context provided," as the claim; the judge correctly reported it unsupported (it isn't a substantive claim), and the real, well-grounded claim after the marker was never checked as its own citation at all. Measured on a 10-question manual baseline run, this pattern hit 4 questions and, in 3 of them, dragged the composite confidence score low enough to trigger `build_fallback_response`'s "I don't have enough information" response on an answer that was otherwise fully correct and grounded.

**Fix is two-pronged: constrain the prompt, and loosen the parser as defense-in-depth** — `GROUNDED_SYSTEM_PROMPT` now explicitly instructs the model to place `[N]` at the end of the sentence/clause it supports, with a correct/incorrect contrast pair, following Anthropic's own prompt-engineering guidance for citation formatting (verified via Context7 against the `anthropics/courses` prompt-engineering material, whose canonical example places the bracket citation at the end of the citing sentence). Since a Haiku-driven judge/generation path (this project's cost-managed default for iterative dev, per the LLM Judge Cost Management section) won't always comply with a prompt instruction, `parse_citations` was also extended to detect a **leading marker** — text preceding a run that is empty or ends with a comma, the structural signature of a lead-in with no claim yet — and in that case scan forward from the marker to the nearer of the next sentence-terminal punctuation (`.`/`!`/`?`) or the next citation run, merging that forward text into the claim instead of returning the lead-in fragment alone.

**Punctuation structure, not a lead-in-phrase blocklist** — The leading-marker signal is "empty or comma-terminated preceding text," not a list of phrases like "according to" or "based on the documentation." A keyword blocklist would need constant expansion as models phrase lead-ins differently and would still miss paraphrases; the comma/empty-text signal generalizes because it targets the *grammatical* shape of a subordinate clause, not its wording.

**Accepted edge case: cascading leading-marker detection** — If a leading marker's forward scan is capped by the next citation run (no terminator in between), that next run's own preceding text is left empty by the scan and it too gets treated as leading, even if the model intended it as an ordinary trailing citation. Closing this fully would require tracking which text spans have already been "claimed" by a prior run's forward scan versus genuinely empty preceding text — meaningfully more state for a heuristic the prompt fix above should make rare in practice. `GROUNDED_SYSTEM_PROMPT` already instructs multi-chunk citations for one claim to be written as a single contiguous run (e.g. `[1][3]`), which is the scenario that would otherwise trigger this; kept as a documented, tested limitation (`test_leading_marker_bounded_by_next_citation_cascades`) rather than engineered away.

**Anthropic's native Citations API considered and deferred** — Context7 lookup surfaced that the Messages API supports a `citations` feature returning structured `cited_text`/`document_index`/character-or-block-location spans, which would eliminate answer-text regex parsing entirely. Adopting it now would mean restructuring `build_grounded_prompt` around per-document `content` blocks and either dropping the OpenAI generation path or maintaining two structurally different citation-extraction mechanisms per provider — a much larger change than this bug warrants, and one that cuts against `citation_verifier.py`'s existing provider-agnostic design (it verifies whatever `[N]`-marked answer text it's given, regardless of which LLM produced it). Left as a documented future direction, not implemented.

## 2026-07-09 — RRF Score No Longer Overwrites `similarity` (reverses 2026-06-30 entry)

**Bug:** `RERANKING_ENABLED=false` is a documented, supported configuration — `HybridRetriever.retrieve()` falls back to slicing `reciprocal_rank_fusion`'s output directly. But the 2026-06-30 decision below had `reciprocal_rank_fusion` stamp every returned hit's `similarity` with the RRF fused score (`weight / (60 + rank)`, max ≈0.0164 for the default 0.7/0.3 weights) — a rank-aggregation artifact, not a `[0,1]` quality signal. Measured on real data (same corpus, same 4 questions, reranker on vs. off): `retrieval_confidence` collapsed from `0.20–0.58` to `0.013–0.016`, and every question — including one the reranked config answered correctly — flipped to `fallback=yes`. `score_confidence` (`src/generation/confidence_scorer.py`) and `build_fallback_response` (`src/generation/fallback_response.py`) both average `hit.similarity` and compare it against `settings.retrieval_confidence_threshold` (default `0.5`), inheriting the `[0,1]` assumption `mean_similarity_confidence`'s own docstring already flagged as violated by RRF-fused hits.

**Fix: `similarity` on returned hits is always the pre-fusion value; the RRF score is internal-only** — `reciprocal_rank_fusion` already built `hits_by_id` (dense-preferred pre-fusion hit per chunk) to work around this exact scale mismatch for its own span's `confidence_score` (see the "Confidence score isolation" entry below) — that pre-fusion lookup is now also the function's actual return value, not just a side computation for tracing. The RRF fused score (`scores` dict) still drives which chunks make the `top_n` cut and in what order; it is simply never written onto a hit's `similarity` field via `dataclasses.replace()` anymore. This is consistent with the wider IR literature's rationale for RRF: rank fusion deliberately discards score magnitude in favor of rank specifically because per-retriever scores (cosine, BM25) aren't calibrated against each other — treating the *fused* score as if it were itself a calibrated magnitude repeats the same category error RRF exists to avoid.

**No new `VectorStoreHit` field, no changes to `score_confidence`/`build_fallback_response`** — Both generation-layer functions were already correct given correctly-scaled input; the mismatch existed only at the retrieval layer where RRF's internal ranking score and the hit's public-facing `similarity` field were conflated. Fixing it there (rather than adding a provenance flag every downstream consumer must check) keeps `VectorStoreHit` a plain, uniform `[0,1]`-scaled-`similarity` contract for every producer — dense, sparse, RRF, and all three rerankers alike.

**Superseded test contract** — `test_similarity_field_set_to_rrf_score` (which asserted the now-fixed leak as intended behavior) is replaced by `test_similarity_field_preserves_pre_fusion_similarity` / `test_similarity_field_never_carries_rrf_score` in `tests/unit/retrieval/test_fusion.py`. `test_no_overlap_merged_and_sorted_descending` — which happened to pass only because the old RRF-score-as-similarity leak made `similarity` monotonic with rank — is replaced by `test_no_overlap_merged_and_sorted_by_rrf_rank`, asserting chunk-id order directly instead.

## 2026-07-09 — Evidence Chain Narrative

**LLM-as-judge over a deterministic string template** — A template mechanically concatenating each span's already-isolated per-span rationale (from `StepQualityVerdict.rationale`) cannot produce genuine cross-span causal reasoning ("this propagated to X, which then..."), because each rationale judges its own span's input→output transformation in isolation and never the relationship between spans. Every other qualitative synthesis already in this codebase (citation verification, answer completeness, step quality, failure categorization) is delegated to an LLM-as-judge behind a `Protocol` + lazy-import factory, so `EvidenceChainJudgeProtocol`/`make_evidence_chain_judge` follow that same shape rather than introducing a one-off template, consistent with the reasoning already used for `FailureCategoryJudgeProtocol`.

**`EvidenceEntry`, not `SpanQualityResult`/`Span`, in the protocol** — `EvidenceChainJudgeProtocol.narrate` takes `list[EvidenceEntry]`, a new, purpose-built, flat dataclass (`step`/`input`/`output`/`score`/`rationale`, all `str`/`int` fields) owned by `evidence_chain.py` itself, rather than reusing `root_cause.py`'s `SpanQualityResult`. This keeps provider implementations decoupled from `root_cause.py`'s dataclasses — the same rationale already documented for why `FailureCategoryJudgeProtocol.classify` takes scalars instead of `RootCauseDiagnosis` directly.

**Evidence-chain reversal — `evaluated_spans` is last-executed-first, `EvidenceChain.evidence` is chronological** — `RootCauseDiagnosis.evaluated_spans` is built by `find_root_cause_span` walking a trace backward, so it's ordered last-executed-first (`root_cause_span == evaluated_spans[-1]`). A narrative that reads "Retrieval... propagated to Generation..." needs to reason forward in execution order, so `build_evidence_chain` reverses `evaluated_spans` before both building the `EvidenceEntry` chain sent to the judge and assembling the returned `EvidenceChain.evidence` — root cause first, last-executed span last.

**Single-field `EvidenceChainVerdict.narrative`, not decision-plus-rationale** — Every other verdict model in this codebase (`StepQualityVerdict`, `FailureCategoryVerdict`) pairs a categorical decision field with a separate `rationale: str` explaining it. `EvidenceChainVerdict` has only `narrative: str`, because there is no separate categorical decision here to justify — the narrative itself *is* the explanation, so a second field would just duplicate it.

**Multi-entry nonce wrapping: one shared nonce, indexed tag names** — Earlier judge prompts (`build_step_quality_judge_prompt`, `build_failure_category_judge_prompt`) each wrap a fixed 2-3 named blocks (`input`/`output`/`quality-rationale`) in nonce-suffixed tags sharing one nonce. `build_evidence_chain_judge_prompt` is the first prompt in this codebase wrapping an *unbounded*, trace-length-dependent number of blocks in one call. Rather than invent new plumbing, it reuses `wrap_with_nonce` unmodified (the helper already accepts an arbitrary `tag` string), passing computed tag names (`f"span-{i}-input"`, `f"span-{i}-output"`, `f"span-{i}-rationale"`) while still sharing one random nonce across every tag in the call — so the "can't forge a boundary" spotlighting defense holds across every entry, not just within its own tag pair. `step`/`score` are safe (a closed `Literal` and a `1-5`-bounded `int`, never untrusted free text) and appear as plain, unwrapped labels, extending the same reasoning already applied to `step` in the step-quality and failure-category prompts to a second scalar type.

**`build_evidence_chain` adds no span of its own** — Same reasoning as `find_root_cause_span`/`categorize_failure`: the judge's own `narrate()` call already gets its own `span("analysis", ...)` from the provider, and the wrapper function has no mid-function logic worth a redundant wrapper span.

## 2026-07-05 — Backward Root-Cause Span Identification

**The root cause is the earliest span in the contiguous unhealthy tail, not simply the last-executed bad span** — Walking `trace.spans` backward and stopping at the *first* judged-bad span would report whichever step happened to run last, which in a cascading failure (e.g. a bad ranking step feeding already-corrupted input into generation and verification) is a downstream symptom, not where the corruption originated. `find_root_cause_span` instead keeps walking backward through consecutive unhealthy spans, remembering each as the current candidate, and only stops once it hits a span scoring *above* `root_cause_quality_threshold` — that span is healthy and marks the boundary of the failing run. The last-remembered candidate (the earliest bad span in that run) is returned as the root cause. This means the healthy boundary span itself is still judged (that's how it's known to be healthy) but never becomes the candidate, and spans before that boundary are never judged at all — only the trace's unhealthy tail costs a judge call, not the whole trace.

**A 1-5 integer score with a single absolute threshold, not a boolean verdict or a relative delta between spans** — `JudgeVerdict`/`CompletenessVerdict` use boolean verdicts because citation support and answer completeness are naturally binary questions. Root-cause scoring needs a graded signal to compare against a threshold (`root_cause_quality_threshold`, default `2`), so `StepQualityVerdict.score` reuses `Span.confidence_score`'s existing 1-5 scale rather than inventing a new range. A relative-delta scheme (comparing each span's score to the previous span's) was considered and rejected: it conflates "this step made things worse than the step before it" with "this step was bad in isolation," and the project spec's own wording ("is this step's output a reasonable transformation of its input") asks about each span's own quality, not a delta.

**The judge prompt is step-aware (`STEP_QUALITY_CRITERIA`), not one generic "is this reasonable" prompt** — A generic prompt risks the judge applying the wrong mental model to a step — e.g. penalizing ranking for "not adding new information," which is exactly what ranking is supposed to do (reorder/cut, not transform content). `STEP_QUALITY_CRITERIA` is a `dict[PipelineStep, str]` giving each step (including `"analysis"`, for type-safety completeness even though real target spans won't carry that step) its own short description of what a reasonable transformation looks like, interpolated into one shared system-prompt template (`ROOT_CAUSE_JUDGE_SYSTEM_PROMPT_TEMPLATE`) rather than five/six separate fixed prompt constants.

**`PipelineStep` gains a sixth value, `"analysis"`, for the step-quality judge's own span** — Every existing judge provider (citation, completeness) wraps its own LLM call in `span(step, ...)` using one of the five original pipeline steps, because that's the step it's embedded in (verification, generation). The root-cause judge doesn't belong to any of those five — it runs later, over an already-completed trace, not as part of producing one. Rather than skip span instrumentation for this judge (leaving it with no prompt/token/latency observability, unlike every other judge), `PipelineStep` was extended with `"analysis"`. The only other place that exhaustively matched the previous five values was one test (`test_all_pipeline_steps_accepted`), updated alongside.

**`find_root_cause_span` itself is not wrapped in `@traced`/`span()`** — Same reasoning as `HybridRetriever.retrieve` being left uninstrumented: each `judge.judge()` call already gets its own `span("analysis", ...)` from the provider, and the walker has no mid-function LLM/arithmetic detail worth a redundant wrapper span — unlike `verify_citations`/`score_confidence`, whose wrapper spans capture short-circuit logic and arithmetic a judge span never sees, the walker's only logic (the threshold comparison) is fully re-derivable from the judged spans' own scores after the fact.

**Standalone unit, no orchestrator, scoped to span identification only** — Same "directly-callable unit" pattern as `verify_citations`/`score_confidence`/`build_fallback_response`: `find_root_cause_span(trace, judge, threshold)` takes an already-loaded `Trace` and judge instance as plain parameters; no orchestrator yet loads a human-flagged trace and calls this automatically. Failure-type categorization (the six labels from the project spec) is implemented separately — see "Failure-Type Categorization" below — and the narrative evidence-chain builder is implemented separately too, in "Evidence Chain Narrative" above; this feature only identifies which span is the root cause.

## 2026-07-09 — Failure-Type Categorization

**Classification is delegated to a second LLM-as-judge call, not derived deterministically from `root_cause_span.step`** — The mapping from a root-cause span's step to a failure category isn't 1:1: a `"generation"`-step root cause could be Extraction Hallucination, Generation Incomplete, or Context Loss, and a fixed `step → category` lookup can't disambiguate between them without re-reading the span's actual input/output. Every other qualitative judgment in this codebase (citation verification, answer completeness, step quality) is already delegated to an LLM-as-judge behind a `Protocol` + lazy-import provider factory, so `FailureCategoryJudgeProtocol`/`make_failure_category_judge` follow that same shape rather than introducing a one-off heuristic.

**`STEP_TO_PLAUSIBLE_CATEGORIES` constrains the judge's choices via the prompt, not via pydantic validation** — Restricting the judge to the category subset that's actually plausible for a given step (e.g. a `"verification"`-step root cause can only be a Citation Error) is stated explicitly in the system prompt as a guardrail against the judge picking an unrelated category. This is prompt-level guidance, not a schema-enforced constraint: `FailureCategoryVerdict.category` accepts any `FailureCategory` value regardless of the span's step, matching how `StepQualityJudgeProtocol` also trusts its judge's output without a second validation pass — a judge that ignores the guardrail produces a verdict `categorize_failure` still returns unmodified, the same trust boundary this codebase already accepts for every other LLM-as-judge call.

**A 7th `"other"` category covers `"ingestion"`/`"analysis"`-step root causes** — The project spec's six-item taxonomy (Retrieval Failure, Ranking Failure, Extraction Hallucination, Citation Error, Generation Incomplete, Context Loss) only names categories for four of `PipelineStep`'s six values. Without a catch-all, `categorize_failure` would have no valid answer for a legitimately-returned `RootCauseDiagnosis` whose root cause is an ingestion-step span — a real possibility `find_root_cause_span` can produce, since it judges whichever step ends up in the unhealthy tail. `FailureCategory` therefore adds `"other"`, and `STEP_TO_PLAUSIBLE_CATEGORIES` maps both `"ingestion"` and `"analysis"` to it exclusively.

**`quality_rationale` is threaded through as an extra classification input, not just `step`/`input`/`output`** — `RootCauseDiagnosis.rationale` (the step-quality judge's own explanation of why the span was unreasonable, from Task 1) is passed to `classify()` as a fourth parameter. The span's raw input/output alone can require re-deriving the same judgment the step-quality judge already made (e.g. distinguishing "this generation output ignores available context" from "this generation output asserts unsupported facts" from raw text); reusing the prior judge's rationale gives the categorization judge a head start instead of discarding that signal.

**`categorize_failure` takes the already-computed `RootCauseDiagnosis`, but the protocol method takes primitives, not the diagnosis object** — `FailureCategoryJudgeProtocol.classify(step, input, output, quality_rationale)` mirrors `StepQualityJudgeProtocol.judge(step, input, output)` in taking scalar/string parameters rather than a domain object, keeping provider implementations decoupled from `root_cause.py`'s dataclasses. `categorize_failure(diagnosis, judge)` is the thin unpacking wrapper that bridges the two — the same shape as how `find_root_cause_span` calls `judge.judge(step=span.step, input=span.input, output=span.output)` rather than passing the `Span` itself.

**`categorize_failure` adds no span of its own** — Same reasoning as `find_root_cause_span`: the judge's own `classify()` call already gets its own `span("analysis", ...)` from the provider, and the wrapper function has no mid-function logic worth a redundant wrapper span (unlike `verify_citations`/`score_confidence`, whose wrapper spans capture short-circuit logic and arithmetic a judge span never sees).

## 2026-07-05 — Trace Persistence — JSON Files + SQLite Index

**`save_trace` never deletes anything in `output_dir`, unlike `src/ingestion/storage.py`'s `save_processed`** — `save_processed` does `shutil.rmtree` then recreates its output directory on every call, because re-ingesting a source file is meant to replace *all* of that file's chunks atomically. Traces have the opposite requirement: every `Trace` is an independent record of one past request, and writing a new one must never delete previously written ones. `save_trace` therefore only does `output_dir.mkdir(parents=True, exist_ok=True)` and writes exactly one `{trace_id}.json` file, touching nothing else in the directory.

**`trace_path` is a column in the SQLite `traces` table even though it wasn't in the original spec's literal column list (`trace_id, timestamp, status, final_score`)** — The whole point of a metadata index is to let a caller find a trace and then open the full record it summarizes; without a stored path back to the JSON file, `get_trace_record`/`list_trace_records` would return metadata with no way to reach the data it's *about*. Reconstructing the path from `trace_id` alone (`trace_output_dir / f"{trace_id}.json"`) was considered and rejected as the sole mechanism — it would require every caller to separately know `trace_output_dir`, when the index row itself can just carry it.

**JSON write succeeding but SQLite indexing then failing propagates the exception uncaught, and the already-written JSON file is not deleted** — Two options existed: treat the pair as one atomic unit (delete the orphan JSON on index failure) or treat the JSON file as the durable source of truth and let a partial index be repaired later. Chose the latter: a missing/stale SQLite row is trivially fixable by re-running `index_trace` against the existing JSON file, but a deleted trace is unrecoverable. This matches the codebase's existing style of raising rather than silently swallowing errors (e.g. `Settings`'s `model_validator`s `raise ValueError` directly rather than logging and continuing).

**No `Protocol`/factory pattern for `src/tracing/index.py`, unlike embedders/vector stores/rerankers/judges** — Every other pluggable component in this codebase (`EmbedderProtocol`, `VectorStoreProtocol`, `RerankerProtocol`, `CitationJudgeProtocol`, `CompletenessJudgeProtocol`) has multiple real implementations selected at runtime via a `make_*(settings)` factory. SQLite is the only storage backend for the trace index — introducing a protocol/factory for a single implementation would be speculative abstraction with no second provider to justify it.

**Raw stdlib `sqlite3`, not an ORM** — No ORM (e.g. SQLAlchemy) was already a project dependency, and one table with four CRUD functions doesn't need one. `_connection()` wraps `with conn:` (which commits/rolls back but does not close the connection — confirmed via the Python 3.11 `sqlite3` docs) in `try/finally: conn.close()`, so every read/write path gets uniform commit/rollback/close semantics without repeating that boilerplate at each call site. Timestamps are stored as `trace.timestamp.isoformat()` text and re-parsed by Pydantic on read (`TraceRecord`), rather than via `sqlite3`'s built-in datetime adapter/converter registration — those defaults are deprecated as of Python 3.12, and this project targets 3.11+, so avoiding them avoids a forward-compatibility trap.

## 2026-07-05 — Confidence Scoring in Spans

**`traced()` gained an optional `confidence_fn` parameter instead of converting every call site to manual `span()`** — Populating `Span.confidence_score` (left unset per the 2026-07-04 entry below) needs each instrumented call to derive a score from its own return value: `confidence_fn(result)` runs after `s.output` is set and its return (`int` 1-5, or `None`) becomes `s.confidence_score`. This covers `DenseRetriever.retrieve`, `SparseRetriever.retrieve`, all three rerankers' `rerank()`, `verify_citations`, and `score_confidence` — six of the seven `@traced` call sites — without touching their bodies, matching `traced()`'s existing role as "sugar over `span()` for the common case where a function's arguments and return value are the whole story."

**`confidence_fn` is typed `Callable[[Any], int | None]`, not `Callable[[T], int | None]`** — The first attempt typed it against `traced()`'s own `T`, reasoning that `T` would unify correctly once the decorator was applied. It doesn't: mypy solves `traced(step, confidence_fn=...)` as a standalone expression before it sees the expression get applied as a decorator, so `T` gets pinned early — either to `Never` (when `confidence_fn` is omitted, as in `build_fallback_response`) or to `confidence_fn`'s own parameter type (e.g. `Sequence[VectorStoreHit]`, wider than the decorated method's actual `list[VectorStoreHit]` return), corrupting the wrapped function's inferred signature and breaking every downstream caller that relied on its exact return type (`RerankerProtocol` conformance, `reciprocal_rank_fusion`'s `list[VectorStoreHit]` parameters). Typing `confidence_fn` as `Callable[[Any], ...]` keeps `T` solvable only from `decorator`'s `func` parameter, exactly as before this feature. Caught by running `mypy src/` after the initial implementation — zero errors before, six after, all traced to this one signature.

**`mean_similarity_confidence` takes `list[VectorStoreHit]`, not `Sequence[VectorStoreHit]`** — Same root cause as above: even with `confidence_fn: Callable[[Any], ...]` on `traced()`, the concrete helper's own parameter type must match what its callers (`DenseRetriever.retrieve`, `SparseRetriever.retrieve`, the three rerankers) actually return, or passing it as `confidence_fn=mean_similarity_confidence` at each call site risks the same over-widening if any future refactor re-links it to `T`.

**`reciprocal_rank_fusion` converted from `@traced` to manual `with span(...)`, uniquely among the six retrieval/ranking call sites** — At the time, RRF's returned `similarity` was the fused RRF score (`~1/60` per weighted-rank term), not a `[0,1]` quality signal — feeding it through `confidence_from_score` would bottom out at confidence `1` on every call regardless of actual retrieval quality (verified by a regression test: `test_confidence_score_uses_underlying_similarity_not_rrf_score`). The fix reused `mean_similarity_confidence` but called it directly, inside the function, on the *pre-fusion* hits (`hits_by_id[cid]`, captured before their `similarity` was overwritten with the RRF score) — mid-function detail no `confidence_fn(result)` hook could reach. **Update (2026-07-09):** the RRF-fused-score-as-`similarity` behavior this paragraph describes was reversed (see the entry at the top of this file) — `result` and `hits_by_id[cid]` are now identical, so the confidence line simplified to `mean_similarity_confidence(result)`. The manual `span()` itself was deliberately left in place rather than reverted back to `@traced`, since re-deriving that conversion is a separate refactor outside the RRF-fix's scope, not something the fix requires.

**`confidence_from_score(value)` is a plain linear map (`round(clamp(value, 0, 1) * 4) + 1`), not a percentile/bucket scheme** — The three continuous 0-1 signals this task maps (mean similarity, citation coverage fraction, `ConfidenceScore.composite`) have no documented distribution to calibrate bucket edges against, so a simple evenly-spaced linear map (0.0→1, 0.25→2, 0.5→3, 0.75→4, 1.0→5) is the least assumption-laden choice. Inputs are clamped to `[0, 1]` first specifically so out-of-range signals (a caller passing a raw RRF score by mistake, or `ConfidenceScore.composite` exceeding 1.0 when caller-supplied weights don't sum to 1) can't violate `Span.confidence_score`'s `ge=1, le=5` constraint.

**`build_fallback_response` gets no `confidence_fn`** — Its own output (`FallbackResponse | None`) is a threshold-gate decision, not a graded quality signal of "its own result" the way retrieval/ranking/generation spans are — the retrieval confidence that drove the gate is already scored on `score_confidence`'s span. Left `None` deliberately, per the already-documented rationale that not every span needs one (see "`Span.confidence_score` is optional" below).

## 2026-07-04 — Span Instrumentation

**`ContextVar`-based sink (`collect_spans()`), not a required `Trace` object per call** — No orchestrator exists yet to assemble one `Trace` per request (every generation-module function is still "a standalone, directly-callable unit" per the entries above). Requiring every instrumented call to build or receive a `Trace` would invent an API shape for an orchestrator that doesn't exist. `collect_spans()` instead yields a plain `list[Span]` that a future orchestrator wraps into a `Trace` itself; outside any `collect_spans()` block, instrumented calls run exactly as before, which is what keeps every pre-existing unit test (none of which set up tracing) passing unmodified.

**`span()` context manager for LLM call sites, `traced()` decorator for everything else** — A decorator can only see a function's arguments and return value, but the LLM prompt, raw response, and token usage a judge provider's `judge()` method produces exist only *inside* the method body, after the API call returns. `span()` is the primitive that supports attaching that mid-function detail (`s.llm_prompt = ...`, `s.token_count = ...`); `traced()` is sugar over `span()` for the common case where a function's arguments and return value are the whole story.

**Token counts extracted via `isinstance` checks, not `try`/`except`** — `unittest.mock.MagicMock` auto-vivifies any attribute access (`mock.usage.input_tokens` returns another `MagicMock`, not an `AttributeError`), so every existing judge-provider test that doesn't explicitly set `.usage` would otherwise leak a `MagicMock` into `Span.token_count` — which fails pydantic validation (`int | None`) and would break tests that never asked for tracing. `_extract_token_count` in each of the four judge provider files checks `isinstance(value, int)` and falls back to `None` rather than assuming the attribute access itself will fail.

**`HybridRetriever.retrieve` is not separately instrumented** — `Span` has no parent/child field, so `Trace.spans` is a flat list. Wrapping the coordinating method in addition to the four leaf calls it makes (`dense.retrieve`, `sparse.retrieve`, `reciprocal_rank_fusion`, `reranker.rerank`) would add a fifth, redundant `retrieval`-step span with no field to express "this one contains those."

**`verify_citations`/`score_confidence`'s wrapper spans nest with their judge spans, unlike `HybridRetriever` — and that's intentional** — `verify_citations` (`@traced("verification")`) calls `CitationJudgeProtocol.judge()` once per in-range citation, and both concrete providers (`AnthropicCitationJudge.judge`, `OpenAICitationJudge.judge`) independently open their own `span("verification", ...)`. Same shape for `score_confidence` (`@traced("generation")`) around `CompletenessJudgeProtocol.judge()`, whose two providers each open their own `span("generation", ...)`. So when a judge is actually called, `collect_spans()` ends up with two spans at the same step value for what looks like one logical operation — exactly the duplication the `HybridRetriever` entry above avoids. The difference: `HybridRetriever` is a pure pass-through with no logic beyond calling four already-instrumented sub-steps, so its outer span would add zero information. `verify_citations` and `score_confidence` each do real work the judge span never sees — `verify_citations` short-circuits out-of-range-index citations *without ever calling the judge*, so for those citations the outer span is the only record that anything happened; `score_confidence` computes `retrieval_confidence` and `citation_coverage` as pure arithmetic that no judge call touches, so only `answer_completeness` ever reaches a judge span. Removing the outer `@traced` (mirroring the `HybridRetriever` fix) would silently drop that information rather than merely de-duplicating it. A future orchestrator building `Trace.spans` from a `collect_spans()` list should expect the same step value (`"verification"` or `"generation"`) to appear more than once per logical call when a judge is invoked — that's not a bug, the two spans carry different information (the wrapper's `input`/`output` cover the whole operation including short-circuits/arithmetic; the judge's span carries the real LLM prompt/token/response detail).

**`Span.confidence_score` left unset by this task** — Populating it requires mapping the continuous 0–1 floats this codebase actually computes (retrieval similarity, `ConfidenceScore.composite`) onto the model's `1-5` int range. No such mapping is specified anywhere in the project docs; inventing one wasn't part of this task's scope, so it's left for whichever future orchestrator decides that conversion. (Superseded 2026-07-05 — see "Confidence Scoring in Spans" above.)

**`traced()` retyped with `ParamSpec`/`TypeVar` before the ninth call site, not left as `Callable[..., Any]` with a `# type: ignore` per site** — `traced()` was originally typed with `Callable[..., Any]`, which is what mypy strict mode considers an "untyped decorator": wrapping any function with it erases that function's real parameter and return types, and mypy flags the decoration itself as an error unless suppressed. This first surfaced instrumenting `DenseRetriever.retrieve` — the first real `@traced(...)` call site after the decorator itself was built — and eight more call sites across Tasks 6–15 were about to apply the same decorator. Accepting a `# type: ignore` at each one would have scattered nine (and growing) suppressions for the same root cause instead of one. `traced()` was retyped using `ParamSpec`/`TypeVar` (PEP 612) — `Callable[[Callable[P, T]], Callable[P, T]]` — so the wrapper preserves the wrapped function's exact signature, fixing the error at its source; every later call site (Tasks 6–15) confirms zero `# type: ignore` comments were needed anywhere in the codebase.

## 2026-07-04 — Trace/Span Data Models

**Plain (non-frozen) pydantic `BaseModel`, not the frozen-dataclass convention used for `FallbackResponse`/`ConfidenceScore`** — Every other judge-free result type in this codebase (`FallbackResponse`, `CitationVerificationResult`) is a frozen `@dataclass`, since none of them are ever handed to an LLM SDK or need to cross a process boundary. `Trace`/`Span` are different: they're the record a future JSON file writer and SQLite indexer will persist, so they need reliable JSON (de)serialization. `ProcessedDocument`/`Chunk` in `src/ingestion/models.py` face the identical requirement and are already plain pydantic models for the same reason — `Trace`/`Span` follow that precedent rather than the frozen-dataclass one.

**Closed `Literal`s for `step` and `status`, not open `str`** — Matches `ChunkingStrategy`'s existing convention (`src/ingestion/models.py`) for enumerations that are fixed by the project spec: the five pipeline steps (ingestion/retrieval/ranking/generation/verification) and three trace statuses (success/failure/degraded) are named explicitly in the spec, so a typo should fail validation immediately rather than silently producing an uncategorizable trace or span.

**`Span.confidence_score` is optional (`1-5` when present), not required** — Not every pipeline step naturally produces a confidence score — an ingestion span has no LLM-judged output to score, while a generation or verification span might. Requiring it on every span would force non-scoring steps to supply a meaningless placeholder value.

**`Trace.status` is required, with no default** — Unlike `Trace.spans` (defaults to `[]`) and `Trace.final_output` (defaults to `None`), `status` has no default. The object represents "the complete record of what happened" per the project spec's own phrasing — callers (the future context manager) must state the outcome explicitly once a request finishes, rather than the model silently assuming an optimistic default like `"success"`.

**Standalone models, not wired into a context manager, decorator, or writer** — Same situation as every Phase 2 generation module before an orchestrator existed: `Trace`/`Span` are usable and fully tested today, but nothing yet constructs a `Span` automatically as pipeline steps run, appends it to a `Trace`, or persists the finished `Trace` to JSON/SQLite. Those are separate, later tasks in the tracing module.

## 2026-07-04 — Graceful Fallback for Low Retrieval Confidence

**Trigger on `retrieval_confidence`, not the composite score** — The composite mixes in citation coverage and answer completeness, either of which can be low for reasons that have nothing to do with whether the right documents were retrieved (e.g. the LLM under-cited a well-grounded answer). The project spec calls out "retrieval confidence" by name for this decision, so `build_fallback_response` takes `ConfidenceScore.retrieval_confidence` as an explicit parameter rather than `ConfidenceScore.composite`.

**No LLM call** — Unlike answer completeness, this decision only needs the similarity scores and metadata already attached to `VectorStoreHit` (mean similarity vs. a threshold, plus title/section/source_path for the summary). Making it a judge call would add latency and cost to answer the same question a threshold comparison already answers deterministically.

**`FallbackResponse` is a frozen dataclass, not a pydantic `BaseModel`** — `JudgeVerdict`/`CompletenessVerdict` are pydantic models specifically because they're passed as `output_format=`/`response_format=` to LLM SDKs' structured-output APIs. `FallbackResponse` is never handed to an LLM, so it follows the codebase's default frozen-dataclass convention instead (matches `CitationVerificationResult`).

**`>=` threshold convention, matching the existing dedup check** — `retrieval_confidence >= threshold` means "confident enough, proceed with generation" (returns `None`). This mirrors `ChromaVectorStore`'s duplicate check (`(1.0 - distances[0]) >= self._threshold`) rather than inventing a new boundary convention for thresholds in this codebase.

**Fallback response is a standalone unit, not wired into a generation orchestrator** — Same situation as citation verification and confidence scoring: no code yet calls an LLM to produce the initial grounded answer, so callers are expected to compute a `ConfidenceScore` first and pass its `retrieval_confidence` in explicitly. It will be composed into an end-to-end `ask()` flow once that orchestrator exists.

## 2026-07-03 — Answer Confidence Scoring

**Boolean `complete`/`incomplete` verdict, mapped to `1.0`/`0.0`, rather than a continuous completeness score** — Matches the existing `JudgeVerdict.supported` boolean pattern from citation verification rather than asking the judge for a 1-5 or 0-1 score directly. A binary verdict is easier for an LLM judge to return consistently and easier to unit-test with canned fixtures than a continuous score whose exact numeric value would otherwise need its own calibration.

**`score_confidence` takes plain `float` weight parameters, not a `Settings` object** — Mirrors `reciprocal_rank_fusion(dense_weight=0.7, sparse_weight=0.3, ...)` in `src/retrieval/fusion.py`: the pure aggregation function stays decoupled from `Settings` and fully testable without constructing one; the future composition-root caller passes `settings.confidence_retrieval_weight` etc. explicitly, same as `HybridRetriever` already does for RRF's weights.

**Citation coverage is `0.0`, not `1.0`, when no citations were parsed** — An answer with zero verified citations has provided zero evidence of grounding, so it should score low on this dimension rather than scoring a vacuous "100% of nothing." This keeps `citation_coverage` interpretable as "how much of what was claimed is actually backed by evidence," not "how much of what little we checked passed."

**No new `pyproject.toml` extras or API keys** — The answer-completeness judge reuses `embed-openai` (`openai>=1.92.0`) and `llm-anthropic` (`anthropic>=0.100.0`), exactly as `CitationJudgeProtocol`'s providers already do. Both features share the same SDKs and the same `OPENAI_API_KEY`/`ANTHROPIC_API_KEY`, so there is no reason to add a parallel dependency surface.

**Confidence scoring is a standalone unit, not wired into a generation orchestrator** — Same situation as citation verification: no code yet calls an LLM to produce the initial grounded answer, so `score_confidence(query, answer_text, hits, citation_results, judge, ...)` takes all of these as plain parameters rather than generating any of them itself. It will be composed into an end-to-end `ask()` flow once that orchestrator exists.

**The "below-threshold → I don't know" fallback from the project spec is out of scope for this feature** — `score_confidence` returns the composite score and its three dimensions; deciding whether a low score should trigger `INSUFFICIENT_CONTEXT_RESPONSE` (already defined in `src/generation/prompts.py`) is left to the not-yet-built orchestrator, consistent with confidence scoring being a standalone unit.

## 2026-07-03 — Citation Verification

**`JudgeVerdict` is a pydantic `BaseModel`, breaking the codebase's usual frozen-dataclass convention for result types** — Every other pipeline result type (`VectorStoreHit`, `Citation`, `CitationVerificationResult`) is a frozen `@dataclass`. `JudgeVerdict` isn't, because both LLM SDKs' structured-output APIs (Anthropic's `messages.parse(output_format=...)`, OpenAI's `chat.completions.parse(response_format=...)`) require a pydantic model class as the schema argument, and the returned parsed object is itself an instance of exactly that class. Defining a separate dataclass and hand-converting on every provider call would add a translation step with no benefit; passing `JudgeVerdict` straight through keeps both providers' `judge()` implementations a single call plus a null-check.

**One judge call per citation, no batching** — `verify_citations` could in principle batch several citation-claim pairs into one LLM call to cut cost and latency. It doesn't, because the project spec's own wording ("send *each* citation-claim pair to an LLM-as-judge") specifies per-pair verification, and per-pair calls keep each verdict's evidence isolated — a single malformed or adversarial chunk can only ever influence its own citation's judgment, not bleed into a neighboring claim's evidence within the same prompt. Batching is a plausible future optimization if judge-call cost becomes a bottleneck, not a correctness requirement today.

**Out-of-range citation indices are rejected before calling the judge, not sent to the LLM to evaluate** — A citation like `[7]` when only 5 chunks were retrieved is unambiguously wrong — there is no chunk 7 to check. Sending it to the judge anyway would cost a real API call to answer a question the pipeline already knows the answer to (unsupported), and would require the judge prompt to handle a "no evidence available" case it doesn't otherwise need. Short-circuiting locally is both cheaper and enforces the grounded prompt's own "do not fabricate citations" instruction mechanically rather than trusting the judge to catch it.

**Citation verification is a standalone unit, not wired into a generation orchestrator** — The codebase has no code yet that actually calls an LLM to produce the initial grounded answer (`src/generation/prompts.py` only builds the prompt). Building that orchestrator was out of scope for this feature — `verify_citations(answer_text, hits, judge)` takes already-generated answer text as a plain parameter, so it's fully testable and usable today, and will be composed into an end-to-end `ask()` flow once that orchestrator exists, rather than this feature reaching backward to build it as a side effect.

**`RuntimeError`, not a bare `assert`, when an SDK returns no parsed structured output** — Both provider `judge()` implementations narrow the SDK's `Optional[ResponseFormatT]` return type before returning it. An `assert` was tried first but rejected in review: Python strips `assert` statements under `python -O`/`PYTHONOPTIMIZE=1`, which would let `None` propagate silently into the caller, later surfacing as an opaque `AttributeError` on `verdict.supported` instead of a clear, actionable error naming the actual failure.

**Nonce-tag wrapping (`wrap_with_nonce`) is shared between the grounded prompt and the judge prompt, not reimplemented** — Both prompts embed untrusted text (retrieved chunk content in the grounded prompt; claim and evidence text in the judge prompt, the latter also chunk-derived) that could contain an attempted prompt injection. Extracting the existing nonce-boundary logic from `build_grounded_prompt` into a standalone `wrap_with_nonce` helper means both call sites get the identical spotlighting defense with one implementation to audit, rather than two independently-written (and potentially divergent) copies.

**`openai>=1.92.0` version floor, bumped from the embedding extra's inherited `openai>=1.30`** — `OpenAICitationJudge` relies on `client.chat.completions.parse(..., response_format=...)`, the stable (non-beta) structured-output API. Version-bisecting PyPI wheels showed this method was only promoted out of `client.beta.chat.completions.parse` into the stable namespace in `openai==1.92.0` (`1.91.0` still lacks it). The repo's prior floor of `1.30` — inherited from the `embed-openai` extra, which only needs the embeddings endpoint — predates that promotion by many releases, so it would have silently allowed installing a version where this provider's `judge()` raises `AttributeError` at runtime. `pyproject.toml` now pins `openai>=1.92.0` for this feature's extra.

## 2026-07-03 — Cohere & Voyage Reranker Providers

**Equality check against the sentence_transformers default, not prefix matching, for reranker model-default resolution** — `make_embedder` distinguishes "the user left this on its default" from "the user chose a provider-specific model" by checking a model-name prefix (e.g. `text-embedding-*` for OpenAI). That trick fails for rerankers: Cohere's model (`rerank-v4.0-pro`) and Voyage's model (`rerank-2.5`) both start with `rerank-`, so a prefix check can't tell which provider a customized `RERANKER_MODEL` was meant for. Instead, `make_reranker` substitutes a provider's own default only when `settings.reranker_model` still exactly equals the sentence_transformers default — the one case unambiguously meaning "untouched" — and passes any other value through verbatim, so an explicit user choice is never silently overridden by a guess.

**Both new providers preserve the hosted API's returned order rather than re-sorting locally** — Cohere and Voyage's rerank endpoints already return results sorted descending by relevance score; both `CohereReranker.rerank()` and `VoyageReranker.rerank()` map results back onto `VectorStoreHit`s by iterating the API response in its returned order. Re-sorting locally would be redundant work that risks introducing a tie-breaking inconsistency with the provider's own ranking.

**No new API keys or `pyproject.toml` extras for reranking** — `cohere_api_key`/`voyage_api_key` and the `embed-cohere`/`embed-voyage` optional extras already exist from the embedding providers, and both rerank endpoints live in the same SDK packages. Adding parallel `rerank_cohere_api_key`-style settings or separate extras would duplicate configuration for no benefit — a user who wants Cohere for either embedding or reranking installs the same extra and sets the same key.

## 2026-07-02 — Cross-Encoder Reranker

**`sentence-transformers` CrossEncoder over LLM-as-judge for the default reranker** — The spec allows either "a small model or LLM-as-judge." A local cross-encoder (`cross-encoder/ms-marco-MiniLM-L6-v2`) requires no API key, no network round-trip, and no per-query cost, matching the project's existing default-to-local-model bias (`EMBEDDING_PROVIDER=sentence_transformers` is likewise the default embedder). LLM-as-judge reranking is left as a documented future provider: it would add per-query LLM cost and latency to every retrieval call and duplicates capability Phase 4's LLM-as-judge root-cause analysis will already exercise.

**`rerank_candidate_pool` (new) vs. `rerank_top_n` (existing, unchanged) — the RRF cutoff and the final answer size are now two separate settings** — Previously RRF's `top_n` was fed directly from `rerank_top_n` (5), so RRF itself performed the final cut and no reranking pass existed. Reusing `rerank_top_n` for the larger candidate pool would have silently changed its meaning for anyone already relying on "top 5" as the final answer size. Instead, `rerank_candidate_pool` (default 20) is a new, additive setting for RRF's output size; `rerank_top_n` keeps its original meaning — the number of chunks that reach generation — whether or not a reranker is installed. A `model_validator` enforces `rerank_top_n <= rerank_candidate_pool`.

**Reranker as an injected, optional dependency on `HybridRetriever`, gated by `reranking_enabled`** — Consistent with `Indexer`'s existing optional-constructor-arg pattern for `embedder`/`vector_store`/`bm25_store`. `HybridRetriever` never imports `make_reranker` itself; composition-root code decides whether to build and pass one. When no reranker is injected, or `reranking_enabled=False`, `retrieve()` falls back to slicing RRF's candidate pool directly — today's exact pre-reranker behavior — so existing callers and tests are unaffected without code changes.

**Cross-encoder scores reuse the `similarity` field on `VectorStoreHit`, via `dataclasses.replace()`** — Same convention as RRF (weighted rank score) and BM25 (max-normalized score) before it: each pipeline stage's current relevance signal overwrites `similarity` on a fresh frozen-dataclass copy rather than introducing a new field per stage. Cross-encoder scores are unbounded and model-dependent (not guaranteed to lie in [0, 1]), same caveat that already applies to raw RRF scores stored in this field today.

## 2026-06-28 — Document Loader

**Simple dispatcher over a plugin registry** — `DocumentLoader.load()` inspects the file extension and calls a private format-specific function. Adding a new format requires one new function and one new branch in the dispatcher. A plugin registry would add indirection for no gain at four supported formats.

**SHA-256 of raw bytes as `doc_id`** — The ID is computed before any processing, so all documents produced from the same file share the same ID regardless of when they were loaded. This makes downstream deduplication a simple equality check on `doc_id` rather than a similarity scan.

**One `ProcessedDocument` per section (Markdown/HTML), per page (PDF), per file (plain text)** — The loader normalises structure, not chunks. Chunking is Phase 1's next step and operates on `ProcessedDocument` objects. Mixing loading and chunking in one pass would make both harder to test and replace independently.

**Section heading extraction is best-effort, not uniform** — Markdown and HTML have reliable structural markers (`#`/`<h1>`–`<h6>`); the loader extracts them. PDF has no reliable heading signals without layout analysis; the loader records `None` and uses the page boundary instead. Plain text has no structure at all. Honesty about what each format can provide is better than fabricating headings.

**Storage mirrors the source path** — Processed output lives at `data/processed/<filename>/page_NNN.json` or `section_NNN.json`. The directory name matches the source filename, making the link between raw and processed files immediately obvious without a database lookup. Re-ingesting overwrites the directory entirely; because `doc_id` is deterministic, downstream consumers detect unchanged content without re-embedding.

**Regex over `markdown-it-py` token stream for heading extraction** — The `markdown-it-py` token stream works well for full rendering but requires tracking parent tokens to identify headings. A single `re.compile(r"^(#{1,6})\s+(.+)$", re.MULTILINE)` over the raw text is simpler, equally correct for well-formed Markdown, and easier to reason about.

## 2026-06-28 — Chunking

**Three switchable strategies via `Settings.chunk_strategy`** — `fixed_size`, `recursive_header`, and `semantic` are controlled by a single env var. The dispatcher in `Chunker.chunk()` selects one strategy per call. All three preserve the full `ProcessedDocument` metadata on each `Chunk`, so downstream consumers don't need the source document.

**`chunk_id = sha256("{doc_id}:{text}")` (content-addressed)** — Identical text from the same document always produces the same `chunk_id` regardless of chunking settings. This makes deduplication a hash lookup rather than a similarity scan, matching the design of `doc_id` in the loader.

**Semantic strategy uses cosine distance, not similarity** — `distances = 1.0 - cosine_similarity`. Breakpoints are sentences where `distance > np.percentile(distances, semantic_breakpoint_percentile)`. Using distance (higher = more different) makes the threshold direction intuitive: "split at the top X% most dissimilar adjacent sentences."

**`chunk_overlap` is silently ignored in semantic mode** — Semantic chunks follow natural topic boundaries; overlap would duplicate sentence content across topic groups, which defeats the purpose. The `chunk_overlap` setting is validated globally (must be < `chunk_size`) but not applied in the semantic path. Callers that set `CHUNK_OVERLAP` with `CHUNK_STRATEGY=semantic` should expect no overlap.

**Lazy OpenAI client instantiation in `_semantic`** — The client is declared `None` and instantiated only when the first multi-sentence document is processed. Single-sentence documents skip the embeddings call and never create the client, so `CHUNK_STRATEGY=semantic` with only short documents incurs no API interaction.

**Cross-field validator: `chunk_overlap < chunk_size`** — `RecursiveCharacterTextSplitter` raises an opaque error when overlap ≥ size. A `model_validator(mode="after")` on `Settings` catches this at startup with a clear message rather than at chunking time.

## 2026-06-29 — Retrieval Indexing

**ChromaDB cosine space via `metadata={"hnsw:space": "cosine"}`** — ChromaDB defaults to L2 distance. Cosine similarity is the correct metric for normalized embedding vectors: two semantically identical sentences embedded separately should score 1.0 regardless of magnitude. Setting the space at collection creation time makes the deduplication formula `(1 - distance) >= threshold` straightforward and scale-invariant.

**`COLLECTION_NAME = "rag_chunks"` as an exported constant** — Other modules (tests, future retrieval query path) need to open the same collection by name. Exporting it from `vector_store.py` gives a single source of truth rather than a magic string repeated across files.

**Deduplication via batched nearest-neighbour query** — All candidate embeddings are passed to `collection.query()` in a single call (`n_results=1`); ChromaDB returns one result set per query embedding. `results["distances"][i][0]` gives the cosine distance from candidate i to its closest stored neighbour. A chunk is a duplicate when `(1 - distance) >= threshold`. This is O(1) round-trips regardless of batch size and is idiomatic to ChromaDB's multi-query API.

**`BM25Okapi` corpus uses `text.lower().split()` tokenization** — Consistent tokenization between index and query is more important than sophisticated tokenization for this domain. Lowercasing normalizes case variations; whitespace splitting is deterministic and dependency-free. The same `_tokenize` function is used for both `add` and `get_scores`, guaranteeing consistency.

**BM25 index persisted as pickled `{chunk_ids, corpus}`, not the `BM25Okapi` object** — Pickling the `BM25Okapi` object would couple the saved state to the installed version of `rank_bm25`. Pickling only the raw corpus and rebuilding on `load()` avoids deserialization errors when the library is upgraded. The rebuild cost is negligible for the corpus sizes expected in Phase 1.

**BM25 persistence path: `Path(settings.chroma_persist_dir).parent / "bm25_index.pkl"`** — Both indexes live under `data/`: ChromaDB at `data/chroma/` and BM25 at `data/bm25_index.pkl`. Co-locating them under the same parent makes backup and migration simple (copy `data/`), and the path is fully derived from the existing `CHROMA_PERSIST_DIR` setting without introducing a new env var.

**Parallel upsert via `ThreadPoolExecutor`, futures collected outside the `with` block** — ChromaDB upsert (I/O-bound) and BM25 `add` (CPU-bound, in-memory) are independent and benefit from concurrent execution. The `with ThreadPoolExecutor` block contains only `executor.submit()` calls; `.result()` collection happens after the context exits. This matters for exception safety: `__exit__` calls `shutdown(wait=True)`, ensuring both threads complete before any result is inspected or `bm25_store.save()` is called. If either future raises, the other thread has already finished and `save()` is skipped — leaving BM25 disk state consistent with ChromaDB on the next `Indexer()` construction.

**`BM25Okapi` requires ≥ 3 documents for discriminative IDF** — With exactly 2 documents, `IDF(term) = log((2 - 1 + 0.5) / (1 + 0.5)) = log(1) = 0` for any term appearing in exactly one document, making all scores 0.0. Tests that assert ranking behaviour use ≥ 3 documents to ensure non-zero IDF values. This is a property of the BM25 formula, not a library bug.

**Optional constructor args for `Indexer` (`embedder`, `vector_store`, `bm25_store`)** — The orchestrator creates its dependencies internally by default, but accepts pre-built instances via keyword-only args. This enables unit tests to inject a mock `Embedder` alongside real `VectorStore` and `BM25Store` instances backed by `tmp_path`, testing sync and dedup behaviour without calling the OpenAI API. The pattern is consistent with how `Chunker` tests mock `openai.OpenAI`.

## 2026-06-30 — RRF Fusion Layer

**RRF constant `k = 60` (Cormack et al. 2009)** — Standard value from the original RRF paper, used universally in production IR systems. Higher k flattens rank differences (top vs. bottom matter less); lower k exaggerates them. 60 is the empirically validated sweet spot for combining ranked lists in document retrieval.

**Weighted RRF, not simple RRF** — `score = weight / (k + rank)` per list, not `1 / (k + rank)`. Allows tuning the dense/sparse balance without changing retrieval k values. Default 0.7/0.3 reflects that dense retrieval generally outperforms BM25 on semantic questions while sparse still adds value for exact keyword matches (function names, config keys, error codes).

**Dense hit metadata priority on overlap** — When a chunk appears in both dense and sparse results, the dense hit's metadata is used. Dense retrieval fetches metadata from ChromaDB directly; sparse hydrates via a second ChromaDB lookup. Using dense metadata avoids a redundant round-trip on overlapping chunks and prefers the richer semantic retrieval path.

**`dataclasses.replace()` to set RRF score on frozen hits** — `VectorStoreHit` is a frozen dataclass. The fusion layer originally updated `similarity` to the RRF score without mutating the original hit via `dataclasses.replace(hit, similarity=rrf_score)`. **Reversed 2026-07-09** (see the entry at the top of this file): the RRF score is no longer written onto returned hits at all — it turned out to be exactly the kind of magnitude RRF's own rank-based design says shouldn't be compared as a quality signal, and downstream code (`score_confidence`, `build_fallback_response`) was silently misreading it as `[0,1]`-scaled when reranking was disabled. `reciprocal_rank_fusion` now returns each selected hit's original pre-fusion object unmodified — no `replace()` call needed for `similarity` at all — while the RRF score still drives selection/ordering internally.

**`HybridRetriever` as thin wiring layer, not algorithm** — `HybridRetriever` contains no retrieval or scoring logic; it reads `Settings`, calls both retrievers, and passes results to `reciprocal_rank_fusion`. Keeps the RRF algorithm independently testable as a pure function (11 unit tests, no mock retrievers) and makes `HybridRetriever` tests focus on wiring only (8 tests, both retrievers mocked).

## 2026-06-30 — Hybrid Retrieval Query Path

**`DenseRetriever` wraps `Embedder` + `VectorStore.query`** — The retriever is a thin stateless facade: embed query → query collection. Keeping embedding and querying separate (rather than adding a `query_text` method to `VectorStore`) means tests can swap `Embedder` for a fixture without touching `VectorStore`, and the same `Embedder` instance is shared across the indexing and query paths with no duplication.

**`SparseRetriever` fetches metadata from `VectorStore`, not `BM25Store`** — BM25 stores only `chunk_ids` and `corpus` (raw tokens) for index construction. Duplicating full chunk metadata (text, title, source_path, etc.) into the BM25 pickle would double storage and create a sync hazard. Instead, `SparseRetriever` calls `VectorStore.get_by_ids(ids)` to hydrate results — one ChromaDB round-trip per query regardless of `k`.

**BM25 scores normalized by max score before returning** — Raw BM25 scores are unbounded and dataset-dependent. Dividing by `max_score` maps results to (0, 1], making the `similarity` field semantically comparable to cosine similarity. This is required for RRF fusion (next step) and for displaying consistent confidence scores across retrieval modes.

**Zero-score guard in `SparseRetriever`** — After sorting, chunks with `score == 0.0` are dropped before normalization. A zero-score chunk means BM25 found no query term overlap — including it would produce `0 / max_score = 0` similarity entries that pollute fusion results. The guard also prevents division-by-zero when all scores are zero (empty index or no term overlap).

**`VectorStoreHit` as the shared result model** — Both `DenseRetriever` and `SparseRetriever` return `list[VectorStoreHit]`. Using a single frozen dataclass means the RRF fusion layer and reranker don't need to know which retriever produced a hit. `dataclasses.replace` in `SparseRetriever` swaps in the normalized BM25 score without mutating the ChromaDB-returned hit.

## 2026-06-30 — Embedding & Vector Store Provider Abstraction

**`Protocol` + factory over inheritance** — `EmbedderProtocol` and `VectorStoreProtocol` are structural (`typing.Protocol`), not abstract base classes. Any object with the right method signatures satisfies the interface without inheriting from anything, which keeps provider modules free of coupling to a shared base class and lets `runtime_checkable` isinstance checks work in tests.

**`sentence_transformers` as the default embedding provider, not `openai`** — Running the pipeline out of the box (`cp .env.example .env && pytest`) should not require an API key. `text-embedding-3-small` (OpenAI, 1536 dims) remains available for production use, but `all-MiniLM-L6-v2` (sentence-transformers, 384 dims, already a base dependency) is what a fresh clone uses by default.

**Collection metadata as the dimension guard, not a config-only check** — Embedding dimensions differ by provider and are fixed on a ChromaDB collection after the first insert. Rather than trust the running config alone, `ChromaVectorStore` writes `embedding_provider`/`embedding_dimensions` into the collection metadata on first creation and compares against it on every later open, raising a clear `ValueError` naming both the stored and configured provider if they diverge. This turns a silent dimension-mismatch corruption into a startup-time error.

**Lazy provider SDK imports inside factory branches, not at module level** — `make_embedder` imports `src.retrieval.providers.embedder_openai` or `embedder_sentence_transformers` only inside the matching `if provider == ...` branch. This means `pip install -e ".[dev]"` (no provider extras) doesn't fail at import time just because `openai` isn't installed, and installing `embed-openai` doesn't force `sentence-transformers` model downloads for users who never select that provider.

**`Embedder`/`VectorStore` kept as backward-compatible aliases via module `__getattr__`** — Existing call sites (and pre-refactor tests) that import the old concrete class names keep working. `__getattr__` resolves them lazily to `OpenAIEmbedder`/`ChromaVectorStore` on first access instead of importing provider SDKs eagerly at module load.

**Model-name guard in `make_embedder`** — If `settings.embedding_provider == "sentence_transformers"` but `settings.embedding_model` still holds an OpenAI-style name (prefix `text-embedding`, the field's own default), the factory substitutes the sentence-transformers default model instead of attempting to load an OpenAI model name locally. Prevents a confusing `sentence-transformers` load error when a user changes only `EMBEDDING_PROVIDER` and forgets `EMBEDDING_MODEL`.
